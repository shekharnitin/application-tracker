To build an automation engine that updates the customer with hyper-accurate progress percentages and realistic ETAs, the system cannot treat the workflow as a fixed linear checklist. Instead, the backend must model term.jpg and health.jpg as a Dynamic State Machine.
When a customer moves down a specific branch (e.g., skipping medicals vs. triggering a physical exam), the engine must dynamically swap the track, recalculating both the milestone denominator and the remaining duration clock.
The Operational Node SLA Weights
To calculate precise percentages and ETAs, the engine assigns standard operational SLAs (processing times) to each individual node from the workflows.
Standard Operational Values
* Node A: Ingestion & PIQA Check = 12 Hours
* Node B: Medical Scheduling & Report Sync = 48 Hours (Conditional)
* Node C: Senior Underwriting Queue = 24 Hours
* Milestone: Final Decision (Approval / Counter Offer / Rejection) = Immediate upon sign-off
Note on Blocker States (Stage 4 - Missing Docs): When an underwriter flags a missing document, the internal SLA clock pauses and a customer countdown timer begins.

🧮 Math Engine Logic for Progress Tracking
To ensure that progress percentages and ETAs scale correctly based on user detours, the engine computes progress dynamically using the following formula:


$$\text{Progress \%} = \left( \frac{\text{SLA Hours of Completed Stages}}{\text{Total SLA Hours of Active Track Path}} \right) \times 100$$
